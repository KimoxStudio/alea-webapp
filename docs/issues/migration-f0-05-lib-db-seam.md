# F0-05: Introduce lib/db seam
Phase: F0 (abstraction seams + reorg, no downtime). Depends on: Pre issues 1-4.
Problem: 15 files call `createSupabaseServerClient()` / `createSupabaseServerAdminClient()` directly from `lib/supabase/server.ts` to obtain a Postgres client. This couples every service-layer file to Supabase directly, so F1 (swap Supabase for Drizzle/Neon) would require touching all 15 call sites again.
Scope: Create `lib/db/` as the single seam for Postgres access, exposing `getDb()` (user-scoped, RLS-respecting) and `getAdminDb()` (admin, RLS-bypassing) as thin wrappers around today's Supabase client factories. Migrate all current call sites -- `lib/server/equipment-service.ts`, `events-service.ts`, `club-events-service.ts`, `tables-service.ts`, `database-time.ts`, `uploads-service.ts`, `saved-games-service.ts`, `users-service.ts`, `rooms-service.ts`, `partners-service.ts`, `library-games-service.ts`, `reservations-service.ts`, `auth-service.ts`, `auth.ts` -- to import from `lib/db` instead of `lib/supabase/server`. `lib/supabase/server.ts` itself is the underlying factory and is left unchanged; it is not a call site. This is a pure indirection seam, not a redesign: no function signatures, exported names, or external behavior change. The existing user-scoped vs admin distinction (root CLAUDE.md "Key conventions") is preserved as-is.
Acceptance criteria: No file outside `lib/supabase/server.ts` and `lib/db/index.ts` imports `createSupabaseServerClient` / `createSupabaseServerAdminClient` directly. `pnpm typecheck` and `pnpm build` succeed with no new errors. Existing service-layer test suites pass unchanged (no test files modified as part of this issue).
Out of scope: `lib/server/auth.ts`'s session-cookie handling (`createSupabaseRouteHandlerClient` / `applyCookies`) and the `app/api/auth/*` route handlers that call it directly -- these belong to F0 issue #6 (`lib/auth/session` seam). No deep redesign of the DB client API (e.g. hiding Supabase-specific types entirely) -- that is F1's job when the real Drizzle/Neon swap happens. No changes to `lib/server` domain regrouping (F0 issue #8).
